# SR Query Builder (dev) アルファテスト手順書

このドキュメントは、`SR Query Builder (dev)` のアルファテスト協力者向けです。
配布 zip（`sr-query-builder-plugin-dev-v0.1.0.zip`）と一緒に共有しています。

---

## 1. これは何のアプリ？

**研究プロトコル（RQ・PICO・組入/除外基準）から、PubMed の検索式を AI で作り、検証・最適化して、他データベース向けに変換するまでを支援する Chrome 拡張**です。システマティックレビュー（SR）の「検索式づくり」工程を対象にしています。

ざっくりの流れ:

1. **プロトコル入力** — RQ・フレームワーク（PICO/PECO/PCC/SPIDER/custom）・組入除外基準を手入力（または `protocol.md` 貼り付け）
2. **シード論文登録** — 「絶対ヒットしてほしい既知の論文」の PMID を登録
3. **ブロック定義** — AI が検索式を 1〜5 個のブロック（`#1`〜`#5`）に分解、ユーザーが承認/編集
4. **ドラフト生成＋検証** — 「生成して検証する」1操作で、ブロックごとのヒット数（line_hits）をライブ表示しつつ、シード論文の捕捉率・MeSH 検証まで自動実行
5. **境界事例の対話的拡張（実験的 / dev）** — 現在の検索式の「外側」を探索し、AI が境界事例を数件提示。include/exclude/maybe をワンクリック判定してシード集合を育てる
6. **編集・エクスポート** — PubMed 検索式 `.md` を出力し、CENTRAL / Embase(Dialog) / ClinicalTrials.gov / ICTRP へ変換

データは **あなた自身の Google Sheets / Drive** に保存されます（プロジェクト=1スプレッドシート）。開発者側はあなたのデータを見られません。

**やらないこと**: スクリーニング、重複除去、PDF取得、nbib自動ダウンロード（ボタンへ誘導するだけ）。

## 2. 事前準備（テスター側）

| 必要なもの | 取得先 / 備考 |
|---|---|
| Google Chrome（最新） | デスクトップ版 |
| Google アカウント | **事前に開発者へ Gmail アドレスを伝えてテストユーザー登録してもらうこと**（後述の理由）。未登録だとログインがブロックされます |
| Gemini API キー（BYOK） | https://aistudio.google.com/apikey で無料発行。拡張のオプション画面に貼り付け |
| NCBI API キー（任意） | https://www.ncbi.nlm.nih.gov/account/settings/ → API Key Management。無くても動くが、PubMed への問い合わせレート上限が緩くなる |

> **重要**: API キーはあなたのブラウザの `chrome.storage` 内にのみ保存され、開発者には送信されません。

## 3. インストール手順（パッケージ化されていない拡張機能の読み込み）

1. Drive から `sr-query-builder-plugin-dev-v0.1.0.zip` をダウンロードし、**展開（解凍）** する
   - 展開したフォルダの直下に `manifest.json` がある状態にする（フォルダが二重になっていたら内側を使う）
2. Chrome で `chrome://extensions` を開く
3. 右上の **「デベロッパーモード」をオン**
4. 左上の **「パッケージ化されていない拡張機能を読み込む」** をクリック
5. 手順1で展開した、**`manifest.json` がある階層のフォルダ**を選択
6. `SR Query Builder (dev)` がカードで表示されればOK
   - 拡張機能IDが **`bckokafmjighegpjiocopkagghppnjld`** になっていることを確認（全員このIDで揃います）
7. ツールバーのパズルピース🧩アイコンからピン留めしておくと便利

> 拡張アイコンをクリック → ポップアップから Google ログイン → プロジェクト作成、と進みます。

### うまくいかないとき
- **「このアプリはブロックされています / 確認されていません」でログインできない** → あなたの Gmail がテストユーザー未登録です。開発者にアドレスを伝えてください（反映に数分かかることがあります）
- **読み込み時に "Manifest file is missing or unreadable"** → フォルダ選択を間違えています。`manifest.json` が直下にある階層を選んでください
- **コードを更新したzipを再配布されたとき** → `chrome://extensions` で当該カードの「🔄 更新（リロード）」を押すか、一度削除して読み込み直し

## 4. テストしてほしいこと

実際に1本、簡単なテーマ（例: 「成人の敗血症に対する早期抗菌薬投与」など手元の研究テーマ）で **home → export まで通し**てみてください。各ステップで下記を見てほしいです。

### 優先度：高（コア機能が動くか）
- [ ] **ログイン/プロジェクト作成**: Google ログイン → 新規プロジェクト作成で、Drive にフォルダと Sheets が作られるか
- [ ] **プロトコル入力 → ブロック生成**: 入力した PICO から AI が妥当なブロックを提案するか。日本語/英語どちらの入力でも壊れないか
- [ ] **ドラフト生成＋検証**: 「生成して検証する」で各 `#N` のヒット数が表示され、登録したシード論文の捕捉率が出るか
- [ ] **エクスポート**: PubMed `.md` が `search_formula.md` 形式（`#N` 行番号、最終行 `#N AND #M`）で出力されるか。他DB変換（CENTRAL/Embase/CT.gov/ICTRP）が出るか

### 優先度：中（体験・正確性）
- [ ] AI の提案（MeSH・シノニム・ブロック改善）は的外れでないか
- [ ] シード捕捉率の数字は、PubMed で同じ式を手で叩いた結果と矛盾しないか
- [ ] **境界事例の対話的拡張（dev）**: 拡張式の外側から出てくる候補は妥当か。include 判定するとシードに追加され、再検証で捕捉率が動くか
- [ ] 履歴（history）から過去バージョンを辿れるか

### 優先度：低（細かいUI/表示）
- [ ] 日本語表記の不自然さ・誤字
- [ ] 画面遷移の詰まり、ボタンの効かない箇所、レイアウト崩れ
- [ ] エラー時のメッセージが分かりやすいか

### 既知の制限（バグ報告不要）
- **`.docx` アップロードは未対応**（「パーサが注入されていません」エラーになる）。プロトコルは手入力か `.md` 貼り付けで
- **LLM は Gemini のみ**（OpenAI/Claude は未対応）
- リロードすると入力途中の下書きは消えます（保存先は Sheets。確定後はそちらに残る）
- ブロック重複・MeSH重複の高度分析（P1ロジック）は未実装

## 5. バグ報告の方法

下記をセットで開発者へ:

1. **何をしようとして / 何が起きたか**（期待 vs 実際）
2. **再現手順**（どのタブ・どのボタン）
3. **スクリーンショット**
4. **コンソールログ**: 不具合画面で `F12`（デベロッパーツール）→ Console タブの赤いエラーをコピー。拡張のバックグラウンド側は `chrome://extensions` → 当該カードの「service worker」リンク → Console
   - ⚠️ ログに API キーやトークンが写っていないか確認してから共有してください
