# ブロック承認 UI ワイヤーフレーム（v0.1）

- **作成日**: 2026-04-17
- **対象画面**: メインビュー `#/blocks`
- **位置づけ**: [requirements.md §4.2 / §11.3](requirements.md) で参照する詳細 UX モック。要件定義時点では ASCII ワイヤーフレームで合意し、実装時に React/Lit 等で具現化

## 1. 画面の役割

`extract-protocol` skill が出力した 1〜5 個のブロックドラフトを、ユーザーが**承認 / 編集**して `Protocol` + `ProtocolBlocks` に書き込むための画面。

**入力**: `extract-protocol` skill の JSON 出力
```json
{
  "framework_type": "pico",
  "research_question": "...",
  "inclusion_criteria": "...",
  "exclusion_criteria": "...",
  "study_design": "RCT",
  "blocks": [
    { "block_label": "Population", "description": "...", "ai_generated": true },
    { "block_label": "Intervention", "description": "...", "ai_generated": true }
  ],
  "combination_expression": "#1 AND #2"
}
```

**出力（保存される行）**:
- `Protocol` に 1 行（version + framework_type + RQ + criteria + block_count + combination_expression + source_*）
- `ProtocolBlocks` に `block_count` 行（version + block_index + block_label + description + ai_generated + note）

## 3. 操作仕様

### 3.1 ブロックの並び替え

- **ドラッグハンドル**: ブロックヘッダ左端に `≡` アイコン（モック上は省略）。マウスでドラッグして順序入れ替え
- **キーボード**: `↑` / `↓` ボタンで 1 つずつ移動。`Alt + ↑` / `Alt + ↓` でも同操作
- 並び替えると `block_index` が自動振り直し → 結合式エリアの `#N` 参照も連動更新（後述）

### 3.2 ブロック追加・削除

- **追加**: ヘッダの「+ ブロックを追加」。最大 5 個まで（5 個到達でボタン disabled）。新規ブロックは末尾に挿入され、`Label` は空、`description` は空、`ai_generated=false`
- **削除**: 各ブロックの「削除」ボタン。確認ダイアログ「ブロック #N を削除しますか？結合式から `#N` 参照も自動的に外されます」→ 確定で削除
- 削除後は残りのブロックの `block_index` を 1 から振り直し、結合式の `#N` 参照を全てリインデックス（§3.5 参照）

### 3.3 ブロック統合

- 各ブロックの「次のブロックと統合」ボタン（最後のブロックでは disabled）
- 押下時、確認ダイアログ「ブロック #N と #N+1 を統合します。Label と Description は両方を残しますか？」
  - 選択肢 1: 両方を結合（Label: `A / B`、Description: 改行で連結）
  - 選択肢 2: #N の内容のみ残す
  - 選択肢 3: #N+1 の内容のみ残す
- 統合後は `block_count` が 1 減り、結合式は `(#N AND #N+1)` を `#N` に置換 → リインデックス

### 3.4 ブロック分割

- ヘッダの `[⋯]` メニューから「このブロックを分割」を選択
- モーダル: Description テキストエリアの中で**カーソル位置で分割**するか、**段落単位で分割**するかを選ばせる
- 分割後は #N と #N+1 の 2 ブロックになり、`block_count` が 1 増える（5 個上限チェック）。結合式は `#N` を `(#N AND #N+1)` に展開 → リインデックス

### 3.5 結合式（`combination_expression`）の編集

- **入力欄**: 自由テキスト。プレビューはリアルタイム更新
- **構文チェック**:
  - 許可トークン: `#1`〜`#5`、`#RCTfilter` 等の名前付きブロック（filter-designer が自動付与）、`AND` / `OR` / `NOT`、`(` / `)`
  - パースエラー時は赤字で「未閉じ括弧」「未定義ブロック `#7`」等を表示し、保存ボタンを disabled
- **リセット**: 「全 AND に戻す」で `#1 AND #2 AND ... #N` に再構築
- **リインデックス挙動**: ブロックの追加 / 削除 / 並び替えが起きると、結合式内の `#N` 参照を自動的に新 index に書き換える（ユーザー編集を尊重し、構造のみ変換）
  - 例: `(#1 AND #2) OR #3` の状態で #2 を削除 → `#1 OR #2`（旧 #3 が新 #2 になる）に変換
  - 自動変換結果はプレビューに「旧: `(#1 AND #2) OR #3` → 新: `#1 OR #2`」と diff 表示し、ユーザーが意図しない変換だった場合に元に戻せるよう Undo を提供

### 3.6 AI 再生成（`block-designer` の単発呼び出し）

- 各ブロックの「このブロックを AI に再生成させる」ボタン
- 押下時にモーダル「再生成の指示（任意）」テキストエリア → 「OK」で `block-designer` skill を該当ブロック単発で再呼び出し
- 結果は diff 表示（旧 description と新 description を並べ）→ accept / reject
- accept 時は `ai_generated=true` で上書き、`LLMApiLog` に `purpose=draft_block` で記録

### 3.7 保存ボタンの 2 種

- **下書きとして保存**: `Protocol` を `version` を新規発番せず**現在のドラフト version を上書き**する（実装上は同じ `version` の追記行で last-write-wins）。シード論文収集（`#/seeds`）には進まない
- **承認してシード論文へ**: 構文チェック OK の場合のみ活性。押下で `Protocol` + `ProtocolBlocks` を新規 `version` で確定保存し、`#/seeds` に遷移（シード論文を先に揃えてから `#/draft` で検索式を生成する流れ）
  - 既存の `Protocol` バージョンを編集している場合は新規 `version` を発番して追記（要件 §3.1 の追記型ポリシー）

## 4. AI 生成バッジの色分け

各ブロックヘッダにステータスを表示：

| バッジ | 条件 | 表示 |
|---|---|---|
| 🤖 AI 生成 | `ai_generated=true` かつユーザー編集なし | グレー文字 |
| ✏️ ユーザー編集 | `ai_generated=false`（ユーザーが何らかの編集を加えた） | 青文字 |
| 🆕 新規追加 | ユーザーが「+ ブロックを追加」で作成、まだ保存していない | 緑文字 |

## 5. 空状態 UI

LLM 抽出に失敗した場合 / 手入力で元テキストが空の場合：

```
┌──────────────────────────────────────────────────────┐
│  ブロックがありません                                │
│                                                      │
│  プロトコル本文がない、または LLM が抽出できません  │
│  でした。手動でブロックを 1 つ追加してください。     │
│                                                      │
│  [+ 最初のブロックを追加]                           │
└──────────────────────────────────────────────────────┘
```

## 6. アクセシビリティ要件

- すべての操作はキーボードのみで完結すること（ドラッグ並び替えは ↑↓ ボタンが代替）
- 各入力欄に `aria-label` を付与
- 結合式の構文エラーは `aria-live="polite"` で読み上げ
- 高コントラストモード（CSS prefers-contrast）対応

## 7. 実装フェーズで詰めるもの

- ドラッグライブラリ選定（dnd-kit 候補）
- 結合式パーサ（独自実装 vs jsep などの軽量 expression parser）
- AI 再生成モーダルのプロンプト指示欄の追加情報（既存ブロックの label / description を含めて再生成プロンプトに自動埋め込みするか）
- ブロック統合時の Description マージ戦略（単純改行 vs LLM で要約マージ）
